# Harness Invariants

Single source of truth for constants and structural invariants shared between
`harness-init` (sets them up) and `harness-sync` (enforces/maintains them).

Both skills MUST reference this file rather than hard-coding values. When a
value changes here, update both skills in the same commit.

---

## File Layout Invariants

| Path | Invariant | Created by | Guarded by |
|------|-----------|------------|------------|
| `CLAUDE.md` | Exactly one line: `@AGENTS.md` | init Step 8 | sync B (`sync-claude-md.sh`) |
| `AGENTS.md` | Target ≤100 lines, hard warn >200 | init Step 3 | sync F (`check-context-size.sh`) |
| `.agents/skills` | Symlink → `../.claude/skills` | init Step 8 | sync E (`symlink-guard.sh`) |
| `backlog.md` | Items use `[ ]` / `[>]` / `[x]` checkboxes under `##` headings | init Step 4 | sync C + D-1 |
| `tasks.md` | Fields: `# Title`, `status:`, `Scope`, `Acceptance Criteria`, `Evaluator Feedback` | init Step 4 | sync C + D-1 |
| `docs/architecture.md` etc. | Referenced from AGENTS.md docs index | init Step 4 | `validate-harness.sh` |

## AGENTS.md Size Policy

| Threshold | Meaning | Who checks |
|-----------|---------|------------|
| 100 lines | **Target** — AGENTS.md is a map, not an encyclopedia | init Step 3, `validate-harness.sh` (PASS ≤100, WARN ≤120, FAIL >120) |
| 200 lines | **Hard warn** — every message reloads this; over 200 burns budget | sync F (`check-context-size.sh`, overridable via `CONTEXT_SIZE_LIMIT`) |

The 100–200 band is a "soft zone": validate flags it, sync stays silent. Fix by
moving content to `docs/*.md` and replacing with a pointer line.

## AGENTS.md Edit Policy (the 4-rule filter)

Embedded verbatim in every AGENTS.md generated by init (under `## Maintenance`)
so *any* session can apply the rule, not only sessions where sync runs.

Update AGENTS.md **only** when ALL of the following are true:

1. Information is not directly discoverable from code / config / manifests / docs
2. It is operationally significant — affects build, test, deploy, or runtime safety
3. It would likely cause mistakes if left undocumented
4. It is stable and not task-specific

**Never add:** architecture summaries, directory overviews, style conventions
already enforced by tooling, anything already visible in the repo, or
temporary / task-specific instructions.

Prefer modifying or removing outdated entries over appending. When unsure, add
a short inline `TODO:` comment rather than inventing guidance.

## Sweep Trigger Policy

`sweep.sh` (installed by init Step 5) performs periodic load-bearing
assessment. It is NOT part of the session-start sync loop — it's heavier and
intentionally sparse.

Acceptable triggers (choose one per project):

- **Manual** — developer runs `bash tools/sweep.sh` between features (default)
- **SessionStart hook** — `.claude/settings.json` with a guard like "skip if
  `tools/.sweep-stamp` is <7 days old"
- **Cron** — weekly CI job via GitHub Actions or `CronCreate`

Record the chosen trigger in `docs/runbook.md` so future sessions know.

## Reconciliation Contract (sync C)

`reconcile-harness.py` moves state between `tasks.md` and `backlog.md`:

| `tasks.md` `status:` | Action | Outcome |
|----------------------|--------|---------|
| `active` / `evaluating` | Leave intact; report sprint title | Nothing moves |
| `done` | Remove matching `[>]` from backlog; delete tasks.md; append CHANGELOG entry if present | Sprint archived |
| `failed` | Revert `[>]` → `[ ]` in backlog with evaluator note; delete tasks.md | Sprint returned to queue |
| (no tasks.md) | Strip orphan `[>]` markers from backlog | Backlog idle state |

For this contract to hold, init MUST create `backlog.md` / `tasks.md` with the
exact schema above.
